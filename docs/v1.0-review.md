# Reader View v1.0 设计文档 Review

针对 `docs/design.md` 的 Review 记录，包含发现的问题、决策过程和结论。

---

## 问题 1：Content Script 在所有页面预注入

**问题描述**：manifest 中 `content_scripts` 配置了 `"matches": ["<all_urls>"]`，每个页面都会注入 content.js 和 reader.css，包括 `chrome://`、空白页等，浪费资源且权限过大。

**方案对比**：

| | 方案 A：按需注入 | 方案 B：保持预注入，缩小匹配范围 |
|---|---|---|
| 注入方式 | 用户触发时通过 `chrome.scripting.executeScript` 动态注入 | manifest 声明，页面加载时自动注入 |
| 资源消耗 | 仅在使用时加载 | 每个页面都加载 JS |
| 权限需求 | `activeTab` + `scripting`，无需 `host_permissions` | 需要 `host_permissions: ["<all_urls>"]` |
| 实现复杂度 | 需要改通信流程 | 简单，保持现有流程 |

**结论**：采用方案 A（按需注入）。去掉 manifest 中的 `content_scripts`，去掉 `host_permissions`，增加 `scripting` 权限，由 background 在用户触发时动态注入。

---

## 问题 2：退出阅读模式恢复原页面的方式

**问题描述**：当前方案用 `cloneNode` 保存 body，退出时用 `innerHTML` 恢复。这会丢失所有事件监听器和 JS 运行时状态，SPA 页面直接废掉。

**方案对比**：

| | 方案 A：隐藏原内容 + 覆盖 | 方案 B：Shadow DOM 隔离 |
|---|---|---|
| Ctrl+F | 正常 | 正常（Chromium 127+） |
| CSS 隔离 | 需要手动处理，原页面全局 CSS 可能渗透 | 天然隔离 |
| 动态元素干扰 | 需要 MutationObserver 监听并隐藏新增元素 | 不受影响，渲染在 shadow 内 |
| 实现复杂度 | 低 | 中（样式需内联或通过 `adoptedStyleSheets` 注入） |
| 原页面状态保留 | 完整 | 完整 |

**结论**：采用方案 B（Shadow DOM 隔离）。使用 `attachShadow({ mode: 'open' })` 创建 shadow root，reader view 渲染在 shadow DOM 内。CSS 天然隔离，不受原页面动态元素干扰，Chromium 127+ 支持 Ctrl+F 搜索。

---

## 问题 3：ESC 退出阅读模式未实现

**问题描述**：6.4 快捷键表格写了 ESC 退出阅读模式，但 content.ts 和 reader.ts 代码中都没有注册 `keydown` 监听。

**结论**：补上 ESC 退出的 `keydown` 监听。在 document 上监听，阅读模式激活时按 ESC 触发退出逻辑。

---

## 问题 4：退出阅读模式的消息流向不通

**问题描述**：reader.ts 中关闭按钮发送 `chrome.runtime.sendMessage({ type: 'TOGGLE_READER' })`，消息到 background，但 background 的 `onMessage` 只处理了 `GET_STATE`，没有处理 `TOGGLE_READER`，消息到不了 content script，退出逻辑走不通。

**结论**：关闭逻辑在 content script 内部闭环。关闭按钮点击 → content script 直接移除 shadow host、恢复原页面显示 → 向 background 发一条状态同步消息更新图标。不需要绕 background 转发。

---

## 问题 5：Popup 和 Reader View 内工具栏功能重复

**问题描述**：Popup 有主题切换、字体设置、导出；Reader View 内的 toolbar 也有同样的功能。文档没有说明两者的定位区别和交互关系——比如在阅读模式下打开 Popup 改了主题，toolbar 的 select 会同步吗？非阅读模式下 Popup 的设置按钮应该是什么状态？

**方案对比**：

| | 方案 A：Popup 做设置，toolbar 做快捷操作 | 方案 B：去掉 Popup，全部放 toolbar | 方案 C：两者都有，通过 storage 同步 |
|---|---|---|---|
| 实现复杂度 | 中（需要明确职责边界） | 低（只维护一套 UI） | 高（需要双向同步） |
| 用户体验 | 入口分散 | 统一，所有操作在阅读模式内完成 | 灵活但容易困惑 |
| 维护成本 | 两套 UI | 一套 UI | 两套 UI + 同步逻辑 |

**结论**：采用方案 B（去掉 Popup）。所有设置和导出功能集中在阅读模式内的 toolbar。点击扩展图标的唯一作用是切换阅读模式。

---

## 问题 6：`host_permissions: ["<all_urls>"]` 不必要

**问题描述**：配合问题 1 的按需注入方案，`activeTab` + `scripting` 权限即可，不需要 `host_permissions`。`<all_urls>` 会在安装时弹出"读取所有网站数据"的权限警告。

**结论**：随问题 1 一并解决，去掉 `host_permissions`。

---

## 问题 7：Markdown 转换方案矛盾

**问题描述**：设计决策中把"内置 Markdown 支持"作为选择 Defuddle 的理由之一，但 popup.ts 又额外引入了 Turndown 做 HTML→Markdown 转换，存在矛盾。

**方案对比**：

| | 方案 A：用 `defuddle/full` 内置 Markdown | 方案 B：用 core bundle + Turndown |
|---|---|---|
| 依赖数量 | 1 个（defuddle/full） | 2 个（defuddle + turndown） |
| Markdown 定制性 | 一般 | 强 |
| 实现方式 | 解析时传 `{ separateMarkdown: true }`，直接拿结果 | 解析后二次转换 |

**结论**：采用方案 A。使用 `defuddle/full` bundle，解析时传 `{ separateMarkdown: true }` 同时获取 HTML（`content`）和 Markdown（`contentMarkdown`），去掉 Turndown 依赖。

---

## 问题 8：Background Service Worker 状态丢失

**问题描述**：Background 用 `Map` 存储标签页的阅读模式状态，但 MV3 的 Service Worker 是短生命周期的，空闲后会被浏览器终止，`Map` 数据全丢。

**结论**：去掉 `Map`，不在 background 维护状态。结合问题 1（按需注入）、问题 4（关闭逻辑内部闭环）、问题 5（去掉 Popup）的决策，background 不再需要维护标签页状态。background 只做两件事：1）监听图标点击 → 向当前 tab 注入/发消息；2）接收 content script 的状态同步消息 → 更新图标。图标状态通过 `chrome.action.setIcon` 设置，由浏览器持久化，不受 Service Worker 生命周期影响。

---

## 问题 9：Reader View 内容存在 XSS 风险

**问题描述**：reader.ts 中 `<div class="reader-body">${options.content}</div>` 直接拼接 Defuddle 输出的 HTML。标题等字段用了 `escapeHtml`，但 content 没有消毒。如果原页面包含恶意内容（如 `<img onerror="...">`），存在 XSS 风险。

**结论**：引入 DOMPurify（~15KB minified），在渲染前对 Defuddle 输出的 HTML 进行消毒。配置白名单保留正文需要的标签，去掉危险属性。Shadow DOM 只隔离样式不阻止脚本执行，不能替代 XSS 防护。

---

## 问题 10：构建工具选择 Webpack 的理由偏弱

**问题描述**：文档说 Webpack 对浏览器扩展生态更成熟，但 2024-2025 年 Vite 的浏览器扩展插件已经相当成熟。Webpack 配置复杂度高，对轻量插件来说偏重。

**方案对比**：

| | Webpack | Vite |
|---|---|---|
| 配置复杂度 | 高（需手动配 ts-loader、CopyPlugin 等） | 低（开箱即用 TS 支持） |
| 构建速度 | 慢 | 快（esbuild 预构建） |
| 扩展插件 | 需要手动拼 manifest、多入口 | `vite-plugin-web-extension` 自动处理 |
| HMR 开发体验 | 需要额外配置 | 原生支持 |
| 项目规模匹配度 | 偏重 | 合适 |

**结论**：采用 Vite 替代 Webpack。项目轻量（去掉 Popup 后只剩 background + content script + CSS），Vite 配置简单、构建快、开发体验好。

---

## 问题 11：CSS 变量挂在 `:root` 上可能与原页面冲突

**问题描述**：Content Script 注入的 CSS 中 `:root` 就是原页面的 `<html>`，如果原页面也用了同名 CSS 变量会互相干扰。

**结论**：随问题 2 一并解决。采用 Shadow DOM 后，CSS 变量定义在 shadow root 内，天然与原页面隔离。

---

## 问题 12：缺少对特殊页面的处理

**问题描述**：没有考虑 PDF 页面、纯图片页面、about:blank、已经是阅读模式的页面等边界情况。

**结论**：在 background 注入前检查当前 tab 的 URL，维护前缀黑名单（`chrome://`、`chrome-extension://`、`about:`、`file://` 等），遇到不支持的页面直接跳过不注入。对于"已经在阅读模式"的情况，向 content script 发消息查询状态——未注入过则消息失败（说明不在阅读模式），已注入则由 content script 返回当前状态。
